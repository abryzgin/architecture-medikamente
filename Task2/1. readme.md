## Комментарии о приватности данных (MVP «Медикаменте»)

### Хранение данных

* **Персональные и медицинские данные** хранятся в PostgreSQL с включённым *Transparent Data Encryption (TDE)* — все файлы таблиц зашифрованы алгоритмом **AES‑256**.
* **Документы** (сканы согласий, результаты анализов) сохраняются в MinIO, где включено объектное шифрование **SSE‑S3** (сервер‑стороннее шифрование) с управлением ключами через **HashiCorp Vault**.
* Секреты и ключи шифрования хранятся и регулярно ротируются в Vault; доступ к ним выдаётся по принципу *минимально необходимого* доступа.

### Сроки хранения и уничтожения

| Категория           | Retention                                     | Алгоритм уничтожения                                  |
| ------------------- | --------------------------------------------- | ----------------------------------------------------- |
| Персональные данные | 5 лет после последнего визита пациента        | Затирание записей + зачистка ссылок (референтов)      |
| Согласия субъектов  | 3 года после отзыва/последнего использования  | Версионированное хранение, удаление старых версий     |
| Логи SIEM           | 1 год «горячие» индексы, 3 года «холодное» S3 | *WORM*‑хранилище; автоматическое удаление по политике |

### Защита данных «в транзите»

* Весь трафик между сервисами идёт по **TLS 1.3**. Для внешних систем применяется **mTLS** (взаимная аутентификация по сертификатам).
* Шлюз API проверяет **JWT**‑токены и применяет политики **RBAC/ABAC**.
* Включены WAF‑правила, лимиты запросов и белые списки IP‑адресов.

### Защита данных «в покое»

* Шифрование блок‑устройств сервера БД через **LUKS** (Linux Unified Key Setup).
* Регулярная ротация master‑ключей в Vault и аудит их использования.
* Логи в SIEM сохраняются в неизменяемом формате **WORM** (Write Once Read Many).

### Мониторинг и аудит

* **SIEM** агрегирует логи от сервисов и Kafka, выполняет корреляцию событий и формирует оповещения.
* Уведомления и доступ к SIEM предоставляются **только администратору ИБ** (роль *securityAdmin*). Сотрудники клиники их не получают.
* Политика хранения логов соответствует требованиям 152‑ФЗ.

## Глоссарий сокращений и терминов

| Сокращение      | Расшифровка                             | Что это значит в контексте проекта                               |
| --------------- | --------------------------------------- | ---------------------------------------------------------------- |
| **API**         | Application Programming Interface       | Программный интерфейс, через который взаимодействуют сервисы     |
| **JWT**         | JSON Web Token                          | Формат токена, в котором передаются роль и атрибуты пользователя |
| **RBAC**        | Role‑Based Access Control               | Контроль доступа на основе ролей (doctor, patient…)              |
| **ABAC**        | Attribute‑Based Access Control          | Контроль доступа на основе атрибутов (clinic\_id, patient\_uuid) |
| **TDE**         | Transparent Data Encryption             | Прозрачное шифрование данных в БД на уровне файлов               |
| **AES‑256**     | Advanced Encryption Standard 256‑bit    | Симметричный алгоритм шифрования с длиной ключа 256 бит          |
| **SSE‑S3**      | Server Side Encryption (S3)             | Объектное шифрование в MinIO/AWS S3                              |
| **LUKS**        | Linux Unified Key Setup                 | Стандарт шифрования дисков в Linux                               |
| **WAF**         | Web Application Firewall                | Фильтр, защищающий веб‑приложение от атак (OWASP Top‑10)         |
| **IP**          | Internet Protocol                       | Сетевой адрес. «Белый список IP» — список разрешённых адресов    |
| **SIEM**        | Security Information & Event Management | Платформа сбора и анализа логов безопасности                     |
| **SOC**         | Security Operations Center              | Центр мониторинга и реагирования на инциденты                    |
| **ELK**         | Elasticsearch + Logstash + Kibana       | Стек для хранения, обработки и визуализации логов                |
| **WORM**        | Write Once Read Many                    | Неизменяемое хранилище: запись однократная, чтение многократное  |
| **mTLS**        | Mutual TLS                              | TLS с взаимной проверкой сертификатов клиента и сервера          |
| **TLS**         | Transport Layer Security                | Протокол шифрования трафика в сети                               |
| **CRUD**        | Create / Read / Update / Delete         | Операции над данными в БД                                        |
| **Vault**       | HashiCorp Vault                         | Система управления секретами и ключами шифрования                |
| **Kafka**       | Apache Kafka                            | Распределённая система потоковой передачи сообщений              |
| **OAuth2**      | Open Authorization 2.0                  | Протокол авторизации для выдачи токенов доступа                  |
| **Kafka Topic** | Канал Kafka                             | Логически разделённый поток сообщений                            |

### Как читать диаграмму

1. **Персоны** зелёным цветом (PlantUML `Person`) — акторы: Пациент, Сотрудник, Администратор ИБ.
2. **Внешние системы** (серый) — Лаборатория, Платёжный провайдер.
3. **Контейнеры** внутри границы `MVP «Медикаменте»` (System\_Boundary) — отдельные приложения/сервисы.
4. **Стрелка** показывает направление взаимодействия, подпись — протокол/тип данных.
5. Комментарии о приватности расположены в блоке `/' … '/` в PlantUML и дублируются в этом файле.
6. Цвет стрелок и контейнеров задаётся автоматически PlantUML; логическая роль видна по названию и описанию.
